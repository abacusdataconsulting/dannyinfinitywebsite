# Cloudflare-native music store with Stripe: a complete architecture guide

**A Cloudflare Workers + D1 + R2 + Stripe stack can power every feature you need — merch sales, digital sheet music delivery, donations, optional accounts, and anonymous tracking — without leaving Cloudflare's ecosystem.** The critical architectural choice is using a single Worker with Hono.js as the API layer, R2 for file storage behind a Worker proxy (never publicly exposed), D1 for all relational data, and Stripe Checkout for every payment flow. Stripe has official GA support for Workers, and every Cloudflare primitive you need — rate limiting bindings, Turnstile, WAF, Web Crypto API — is production-ready today. Cloudflare now officially recommends Workers over Pages for new full-stack projects, with Workers Static Assets handling the frontend for free.

---

## How all the pieces fit together

The architecture is a **single Worker deployment** serving both static frontend assets and API routes. Hono.js handles routing, middleware, and typed bindings. The Worker talks to D1 for relational data, R2 for file storage, and Stripe's API for all payment processing. No other infrastructure is needed.

```
┌─────────────────────────────────────────────────────┐
│  Cloudflare Edge (Single Worker Deployment)          │
│                                                      │
│  Static Assets ─→ SPA (React/Svelte/etc.)           │
│  /api/*        ─→ Hono.js Router                    │
│    ├─ /api/auth/*       → Auth middleware + D1       │
│    ├─ /api/products/*   → Product catalog (D1)      │
│    ├─ /api/checkout/*   → Stripe Checkout Sessions   │
│    ├─ /api/webhooks/*   → Stripe webhook handler     │
│    ├─ /api/donate       → Donation flow (Stripe)     │
│    └─ /api/download/*   → Signed URL validation → R2 │
│                                                      │
│  Bindings: D1 (SQL) · R2 (Files) · Rate Limiter     │
└─────────────────────────────────────────────────────┘
         ↕                    ↕
    Stripe API          Cloudflare R2
  (payments, webhooks)  (PDFs, images)
```

**Hono.js is the strong framework choice** — created by a Cloudflare employee, purpose-built for Workers, ultra-lightweight (~14KB), with built-in middleware for CSRF, CORS, secure headers, JWT auth, and Zod validation. Cloudflare officially recommends it. The `wrangler.jsonc` configuration binds D1, R2, rate limiters, and secrets into a single typed `Env` interface generated by `wrangler types`.

The recommended project structure is a monorepo with `src/worker/` containing Hono routes split by domain (auth, products, orders, downloads, webhooks, donations), middleware (auth, Turnstile verification, logging), and `src/frontend/` for the SPA. The Cloudflare Vite plugin (`@cloudflare/vite-plugin`) runs the actual `workerd` runtime locally, ensuring dev/prod parity.

## Stripe integration runs natively in Workers

Stripe's Node.js SDK has **official, GA support for Cloudflare Workers** — jointly announced by both companies. The only required configuration is using the Fetch-based HTTP client instead of the default Node.js one:

```typescript
import Stripe from 'stripe';

const webCrypto = Stripe.createSubtleCryptoProvider();

function getStripe(env: Env) {
  return new Stripe(env.STRIPE_SECRET_KEY, {
    httpClient: Stripe.createFetchHttpClient(),
    apiVersion: '2024-06-20',
  });
}
```

As of `stripe-node` v11.10+, no `node_compat` flag is required for Stripe itself. The official template lives at `stripe-samples/stripe-node-cloudflare-worker-template` on GitHub. Use **restricted API keys** in production — grant only the permissions your Worker needs (Checkout Sessions: write, Webhooks: read).

Store `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET` via `wrangler secret put` — these are encrypted at rest by Cloudflare. For local development, use a `.dev.vars` file (gitignored). Never put secrets in `wrangler.toml` `[vars]`.

### Three payment flows from a single Worker

**Physical merch** uses Stripe Checkout with `shipping_address_collection` and `shipping_options`. The webhook records the order in D1 with shipping details for manual fulfillment. **Digital sheet music** uses Checkout without shipping — the webhook creates a purchase record in D1 and generates a time-limited HMAC-signed download URL. The success page or email delivers this link. **Donations** use Stripe's first-class "customer chooses price" model with `custom_unit_amount` on a Price object (minimum, preset, and maximum amounts) and `submit_type: 'donate'` on the Checkout Session — Stripe's UI handles the amount picker natively.

For the **MVP free-download-with-donate-button**, the simplest path is a **Stripe Payment Link** created in the dashboard with "Customers choose what to pay" pricing. This requires zero server code — just an `<a>` tag below each free download. You still receive `checkout.session.completed` webhooks for tracking. When you need more control (custom presets, analytics), upgrade to a Worker endpoint that creates Checkout Sessions with `price_data` and validates amounts server-side.

### Webhook signature verification uses the async Web Crypto path

The critical detail: Stripe's synchronous `constructEvent()` uses Node.js `crypto.createHmac()`, which doesn't exist in Workers. You **must** use `constructEventAsync()` with `Stripe.createSubtleCryptoProvider()`:

```typescript
const event = await stripe.webhooks.constructEventAsync(
  body,                      // raw request body as string
  sig,                       // stripe-signature header
  env.STRIPE_WEBHOOK_SECRET, // whsec_...
  undefined,                 // tolerance (default 300s)
  webCrypto                  // SubtleCryptoProvider
);
```

A known gotcha from the Cloudflare blog's original sample: calling `await request.text()` twice causes "Body has already been used" errors. Always store the body in a variable first. For idempotent processing, check D1 for an existing fulfillment record keyed by `session.id` before processing, and use `ctx.waitUntil()` to return `200` to Stripe immediately while handling fulfillment asynchronously — this prevents Stripe timeout retries.

## Authentication as a progressive, optional layer

The site works fully for anonymous users. Account creation is offered post-purchase or when accessing elevated features like download history. This "progressive authentication" pattern avoids the **24% cart abandonment** rate caused by mandatory registration.

**For custom auth, use JWT with Web Crypto API.** The `@tsndr/cloudflare-worker-jwt` library is lightweight, zero-dependency, and Workers-native. A hybrid approach works best: short-lived JWTs (15–30 minutes) for stateless request authentication, backed by D1-stored refresh tokens for persistence and revocation. Transport tokens in `HttpOnly; Secure; SameSite=Strict` cookies.

Third-party options that work in Workers include **Clerk** (`@clerk/backend` built for V8 isolates) and **Better Auth** (has a dedicated `better-auth-cloudflare` package with D1/KV/R2 support and a Hono example). **Lucia Auth is deprecated as of March 2025** — the maintainer converted it to educational documentation. Its code patterns for session tables remain excellent reference material, but don't build on the unmaintained library. For admin routes, layer **Cloudflare Access** (Zero Trust) on top of your custom auth — it's designed for internal access control, not public user registration.

### Password hashing within Workers' constraints

**PBKDF2 via Web Crypto API** is the simplest production-ready option — it's natively available with `crypto.subtle.deriveKey()`. Use **100,000+ iterations with SHA-256** and a 16-byte random salt. For stronger security, use **Argon2id via a Rust Worker service binding** — the `glotlabs/argon2-cloudflare` package provides a pre-built Rust Worker callable with ~2ms overhead and no network hop. Standard `bcryptjs` works but consumes hundreds of milliseconds of CPU time, risking timeouts on the Bundled plan. You need the **Workers Standard plan** (30s CPU limit) for any password hashing under load.

### Anonymous tracking that merges cleanly into accounts

On first visit, assign an anonymous session token (`crypto.randomUUID()`) as an `HttpOnly; Secure; SameSite=Strict` cookie, with a corresponding row in an `anonymous_sessions` D1 table. No fingerprinting — cookies only. If the cookie is cleared, a new session starts. This is acceptable and privacy-respecting.

When a guest purchases, Stripe collects their email. Store the purchase linked to both `anonymous_session_id` and `email`. Post-purchase, the confirmation page offers: "Create a password to access your downloads anytime" — a single field, since you already have their email. On account creation, a D1 batch operation links all prior purchases and downloads:

```sql
UPDATE purchases SET user_id = ? WHERE (anonymous_session_id = ? OR email = ?) AND user_id IS NULL;
UPDATE download_tokens SET user_id = ? WHERE anonymous_session_id = ? AND user_id IS NULL;
DELETE FROM anonymous_sessions WHERE id = ?;
```

Session cookies for cart/purchase functionality are exempt from GDPR consent requirements under the ePrivacy Directive. However, set a TTL on anonymous sessions (30 days) and implement a deletion endpoint for data subject requests.

## Secure digital delivery through R2 with Worker gatekeeper

**Never expose your R2 bucket publicly.** Every file request flows through your Worker, which validates access before streaming content. R2 bindings (`env.R2.get(key)`) are injected at runtime with no exposed credentials — no API keys, no account IDs in URLs.

R2's pricing is exceptionally favorable for a small music site: **10 GB free storage, 10 million free reads/month, and zero egress fees — always**. A catalog of 500 PDFs averaging 5MB each (~2.5 GB) with 5,000 downloads/month fits entirely in the free tier.

### HMAC-signed download URLs with time expiry

Rather than R2's S3-compatible presigned URLs (which expose account IDs and require `aws4fetch`), use **custom HMAC-SHA256 signed tokens** via Web Crypto API. The token encodes `purchaseId:fileKey:expiry`, signed with a secret. The download endpoint verifies the signature, checks the purchase record in D1, validates download count limits, increments the counter atomically with `db.batch()`, and streams the file from R2.

The download validation flow performs five checks: (1) HMAC signature valid, (2) token not expired, (3) purchase exists in D1, (4) download count below maximum (default 5), (5) file exists in R2. Only then does the Worker stream the response with `Content-Disposition: attachment`, `Cache-Control: no-store`, and the appropriate MIME type.

### PDF watermarking works in Workers with pdf-lib

The `pdf-lib` library is pure JavaScript with no native dependencies — it runs in Workers. Watermark every paid download with the buyer's email and order ID as a light footer and optional semi-transparent diagonal overlay. This creates accountability without harming the reading experience. For large PDFs where on-the-fly watermarking exceeds CPU limits, pre-watermark at purchase time and store the personalized copy in R2 as `watermarked/{purchaseId}/{fileKey}`.

**For free content**, serve directly from R2 through the Worker with aggressive caching (`Cache-Control: public, max-age=86400`) and use the Workers Cache API to minimize R2 read operations. Use `Content-Disposition: inline` so free samples render in-browser rather than downloading.

## D1 security model and schema design

**All D1 data is encrypted at rest using AES-256-GCM**, managed by Cloudflare's internal KMS — no configuration needed. Data in transit uses TLS. Cloudflare maintains SOC 2 and ISO 27001 certifications. For extra-sensitive PII fields, add application-level AES-GCM encryption using Web Crypto API with a key stored as a Worker Secret.

**Always use prepared statements with `.bind()`** — this is the single most important D1 security practice. D1 follows SQLite parameter binding conventions, and `.bind()` prevents SQL injection by design. Never use string interpolation in queries. Use `.batch()` for atomic multi-statement operations (D1 doesn't support `BEGIN TRANSACTION`). Use `db.exec()` only for one-shot migration scripts, never with user input.

What to store in D1: user profiles (with hashed passwords), sessions, product catalog, purchase records, download tokens, donation records. What to **never** store: raw credit card data (Stripe handles all PCI-scoped data), plaintext passwords, API keys. Reference Stripe data by `stripe_customer_id` and `stripe_payment_intent_id` — keep Stripe as the source of truth for payment details.

The core schema spans seven tables: `users`, `sessions`, `anonymous_sessions`, `products` (with a `type` enum for merch/digital/sheet_music and an `r2_key` for digital files), `orders` (linked to Stripe payment intent IDs), `order_items`, and `download_tokens` (with download counting and expiry). Index on foreign keys, email, slug, and token columns. D1 supports **Time Travel** — built-in point-in-time recovery to any minute within the last 30 days.

## Security hardening across every layer

**CSRF protection** uses Hono's built-in `csrf()` middleware, which checks both `Origin` and `Sec-Fetch-Site` headers. Combine with `SameSite=Strict` cookies. The Stripe webhook endpoint skips CSRF checks — it uses Stripe signature verification instead.

**Rate limiting** operates at two levels. The **Workers Rate Limiting binding** (GA since September 2025) provides per-location counters with near-zero latency — configure `LOGIN_LIMITER` at 5 requests/60 seconds and `API_LIMITER` at 100/60 seconds in `wrangler.jsonc`. The `@elithrar/workers-hono-rate-limit` package integrates this cleanly as Hono middleware. **Cloudflare WAF rate limiting rules** add a second layer at the edge — configure 4 requests/minute with Managed Challenge on login/register endpoints.

**Input validation** uses Zod with `@hono/zod-validator` — validate all request targets (json, form, query, param) with typed schemas. Add Hono's `bodyLimit` middleware (1MB) to prevent oversized payloads. **Secure headers** come from Hono's `secureHeaders()` middleware, which sets HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and Content-Security-Policy. The CSP must whitelist `js.stripe.com` and `challenges.cloudflare.com` for Stripe Elements and Turnstile.

**Cloudflare Turnstile** protects sensitive forms (login, registration, checkout, donation) against bots. Server-side verification calls `https://challenges.cloudflare.com/turnstile/v0/siteverify` with the token and IP. Tokens are single-use and expire after 5 minutes. Deploy the **Cloudflare Managed WAF Ruleset** immediately for protection against SQLi, XSS, and RCE — it's maintained by Cloudflare's security team and regularly updated. DDoS protection is automatic on all plans.

**CORS** should use specific origins (`['https://yoursite.com']`), never wildcards for authenticated endpoints. Set `credentials: true` when using cookies. Register CORS middleware before routes in Hono.

## Deployment, environments, and operational patterns

Use **environment-specific configurations** in `wrangler.jsonc` with separate D1 databases and R2 buckets for staging and production. Bindings are not inherited between environments — declare them explicitly per env. Secrets are set per-environment with `wrangler secret put --env production`.

Deploy via **GitHub Actions** using `cloudflare/wrangler-action@v3`, with branch-based environment selection (main → production, staging branch → staging). Run D1 migrations with `wrangler d1 migrations apply --local` first, then `--remote`. Enable Workers Observability (logs and traces) before going to production.

Key operational practices: keep `compatibility_date` current, generate types with `wrangler types` rather than hand-writing the `Env` interface, stream request/response bodies to stay within the **128 MB memory limit**, use bindings (not REST APIs) for all Cloudflare services, avoid global mutable state (isolates reuse across requests), and always `await` or `waitUntil()` every promise — floating promises cause silent data loss.

## Conclusion

This architecture keeps everything within Cloudflare's edge network except Stripe API calls, minimizing latency and operational complexity. The three most consequential decisions are: **(1)** using `constructEventAsync` with `Stripe.createSubtleCryptoProvider()` for webhook verification — the synchronous path silently fails in Workers; **(2)** never exposing R2 publicly and instead routing every download through Worker-based access control with HMAC-signed tokens; and **(3)** building progressive authentication where guest purchases link to accounts via email, not session ID, so the merge works even if cookies are cleared between purchase and signup.

The MVP path is fast: Stripe Payment Links for donations (zero backend code), Stripe Checkout Sessions for purchases (one Worker endpoint), a webhook handler for fulfillment, and HMAC-signed R2 downloads. Authentication, watermarking, and advanced rate limiting are additive layers you can ship iteratively. Start with the Workers Standard plan ($5/month) to avoid CPU time limits during password hashing and PDF watermarking operations.
